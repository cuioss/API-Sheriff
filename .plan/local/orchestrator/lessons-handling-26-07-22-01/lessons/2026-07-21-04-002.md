# Deep-lane outline folded away an explicitly-requested wiring deliverable - surfaced as execute-time task infeasibility

The original request for this plan explicitly named route-table **terminal-action
materialization** (the data-plane wiring that makes an asset-serving route actually
serve). The phase-3 (deep-lane) outline under-scoped that data-plane wiring and folded
the terminal-action materialization away into an abstraction, so no deliverable carried
it forward. The gap stayed invisible through phase-4 planning and only surfaced at
execute time as a **TASK-015 infeasibility** — the task could not be completed because
the wiring it depended on had never been scoped as work.

## Rule

- When the request names a **concrete wiring/materialization deliverable** (a specific
  data-plane hop, a producer→consumer connection, a terminal action that must actually
  fire), the outline MUST carry a deliverable that owns that wiring. Do not abstract a
  named, concrete integration point away into a higher-level umbrella deliverable — the
  umbrella hides whether the wiring is actually planned.
- An outline is only complete when every concrete, execution-visible item in the request
  maps to a deliverable. "Covered by the general design" is not coverage for a named
  wiring step.

## Prevention

At phase-3 (especially deep lane), cross-check the request's concrete nouns
(route-table, terminal action, producer, alias collection) against the deliverable list
before closing the outline. A concrete request item with no owning deliverable is a
scope hole that phase-4 planning cannot repair — it becomes an execute-time infeasibility
with a task that has no path to done.

## Impact

Cost a TASK-015 execute-time stall and rescoping on this plan. Recurs on any deep-lane
outline that summarizes concrete data-plane wiring into an abstraction instead of a
deliverable.
