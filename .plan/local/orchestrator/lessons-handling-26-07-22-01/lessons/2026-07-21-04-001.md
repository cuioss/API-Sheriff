# AssetConfig native reflection gap: JVM-green but native-red - register reflectively-bound config classes for GraalVM

A new configuration record (`AssetConfig`) bound and validated green under the JVM
test suite and the local quality gate, but the **native** image build went red in
CI: GraalVM could not instantiate/populate the class because it was never registered
for reflection. The JVM path never exercises the native reflection metadata, so the
gap is structurally invisible to every JVM-mode gate — `verify -Ppre-commit` and
`verify` both pass while the native executable is broken.

## Rule

- When you add a config class (or any DTO) that Quarkus/Jackson/SmallRye Config binds
  **reflectively**, register it for native reflection — `@RegisterForReflection` on the
  type (or a `reflect-config.json` / `ReflectiveClassBuildItem` entry) — in the SAME
  change that introduces the class. Do not wait for CI to discover the omission.
- A green JVM build is NOT evidence the native image works. Reflection, resource, and
  serialization registration gaps are native-only failure modes that no JVM-mode gate
  can catch.

## Prevention

Treat "JVM-green" and "native-green" as two independent gates on any change that adds
a reflectively-bound type. When touching config/DTO records, either run the native
build locally (`./mvnw clean install -Pnative -pl api-sheriff -am -DskipTests`) before
push, or explicitly review reflection registration as part of the change so the native
round is not the first place the gap surfaces.

## Impact

This was one of three consecutive CI re-settles on this plan where a locally-green
gate certified a broken artifact. Every new reflectively-bound config/DTO type in
api-sheriff is exposed to this class until native registration is part of the
authoring checklist.
