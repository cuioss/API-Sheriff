# baseline-reconcile merge-tree probe reports conflict_count=0 on git-version merge_tree_error - version-detect and fail closed

The orchestrator's baseline-reconcile classifier probes whether a plan branch conflicts
with the evolved baseline by shelling out to `git merge-tree`. On this box the probe was
built for one `git merge-tree` CLI form but the installed git only accepts the other, so
the invocation failed with:

```
merge_tree_error: "usage: git merge-tree <base-tree> <branch1> <branch2>"
```

The classifier treated that errored probe as a successful "no conflicts" result and
reported `conflict_count=0`. Ground-truth `git merge-tree` (run manually with the form the
installed git accepts) showed real conflicts. So the reconcile classifier emitted a
false-green "clean landing" signal while the branch actually conflicted — a failed probe
was silently read as a passing probe.

Two independent `git merge-tree` surfaces exist across git versions:

- Older: `git merge-tree <base-tree> <branch1> <branch2>` — prints a merge preview diff.
- Newer: `git merge-tree [--write-tree] <branch1> <branch2>` — writes the merged tree and
  reports conflicts via exit code / `--name-only` output.

Assuming one form without detecting the installed git's supported syntax makes the probe
non-portable across developer/CI environments.

## Rule

- Version-detect the `git merge-tree` CLI form (or prefer the `--write-tree` form and fall
  back on a usage error) before parsing its output — do not hard-code a single argument
  shape that a differently-versioned git rejects.
- Fail CLOSED on a probe error. A `merge_tree_error` (non-zero exit, `usage:` on stderr, or
  unparseable output) MUST NOT be collapsed to `conflict_count=0`. Treat an errored or
  unparseable conflict probe as "conflict status UNKNOWN / assume conflicts present," so the
  reconcile step surfaces the ambiguity instead of certifying a clean landing that was never
  actually verified.

## Prevention

This is the general false-green-on-degenerate-input hazard (cf. lesson 2026-07-18-11-001):
a gate that derives pass/fail from a subprocess must distinguish "probe ran and found zero"
from "probe failed to run." Add a fixture test that feeds the classifier a `merge_tree_error`
stderr and asserts the outcome is NOT `conflict_count=0` / not "clean." Log the raw
`git merge-tree` stderr on any non-zero exit so a version mismatch is diagnosable rather than
silently swallowed.

## Impact

An unreliable `conflict_count=0` from a broken merge-tree probe lets the orchestrator's
baseline-reconcile certify a conflicting branch as clean, corrupting landing/merge decisions.
Recurs on any box whose installed git supports only one `git merge-tree` CLI form while the
classifier assumes the other.
