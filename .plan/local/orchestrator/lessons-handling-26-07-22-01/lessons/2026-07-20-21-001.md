# API-Sheriff marshal.json: documentation domain skills_by_profile lacks an implementation key — doc-only tasks resolve zero raw skills and rely on persona augmentation; reconcile via marshall-steward

`marshal.json`'s architecture inventory for the API-Sheriff `documentation` module
exposes only a `skills_by_profile.documentation` key. Solution-outline deliverables
that touch pure-prose paths (e.g. `doc/**/*.adoc`) are consistently authored with
`profile: implementation` (the canonical profile name used across every other
module), never `profile: documentation`. At phase-4-plan task-creation time this is
a genuine key miss — `module.skills_by_profile.implementation` is empty/missing for
the `documentation` module — so the architecture-resolved default skill set is `[]`
for every doc-only task. The task does not end up unskilled only because the
Persona-Skill Augmentation step (phase-4-plan Step 5) separately resolves
`persona-implementer` against the deliverable's `domain: documentation` field and
that resolution happens to surface the right `pm-documents:*` skills
(`ref-asciidoc`, `ref-documentation`, `ref-narrative-styles`, `ref-svg-diagrams`) —
but this is a lucky compensating path, not a designed one, and it silently produces
a Q-Gate `skill_resolution` warning finding on every doc-only task in every plan
that touches `doc/`.

## Rule

Either (a) the `documentation` module's `skills_by_profile` map in `marshal.json`
should expose the same three canonical profile keys every other module uses
(`implementation` / `module_testing` / `verification`) — aliasing `implementation`
to the existing `documentation` skill list — so architecture-resolved defaults are
populated directly and the phase-5 skill-resolution WARNING (and its Q-Gate finding)
stops firing; or (b) phase-3-outline should be taught to emit `profile: documentation`
for doc-only deliverables against modules that declare a `documentation` profile key,
so the outline-declared profile matches what the architecture inventory actually
models. Option (a) is preferred: it fixes every existing and future plan without
requiring outline-authoring changes, and keeps the profile vocabulary uniform
(`implementation`/`module_testing`/`verification`) across the whole planning
pipeline.

## Prevention

When `marshall-steward` next runs its config-reconciliation / upgrade pass over
`marshal.json`, add an `implementation` alias (or a full profile entry) under the
`documentation` module's `skills_by_profile` block mirroring the existing
`documentation` key's skill list, so the architecture query resolves defaults
directly instead of relying on the persona-augmentation fallback.

## Impact

Every plan with documentation-only deliverables against the `documentation` module
hits this gap. It is currently silent-but-harmless (persona augmentation back-fills
the doc skill set), but it manufactures a spurious Q-Gate `skill_resolution` warning
finding per doc-only task that has to be triaged away instead of never firing.

## Recurrence — 2026-07-21 (protocol-processors-ws-grpc)

Recurred identically across all 5 documentation-module, `profile: implementation`
deliverables in this plan (deliverables 1, 2, 3, 13, 14 — TASK-001, TASK-002,
TASK-003, TASK-016, TASK-017). Each task's architecture-resolved base skill set was
empty (`module.skills_by_profile.implementation` missing on the `documentation`
module) and a WARNING + Q-Gate triage finding was logged per phase-4-plan Step 5's
documented fallback; each task's final `skills[]` was populated only via the
Persona-Skill Augmentation step resolving `persona-implementer` against
`domain: documentation`, landing the same 6-skill set
(`plan-marshall:persona-plan-marshall-agent`, `plan-marshall:ref-code-quality`,
`pm-documents:ref-asciidoc`, `pm-documents:ref-documentation`,
`pm-documents:ref-narrative-styles`, `pm-documents:ref-svg-diagrams`) on every
affected task. No functional impact on this plan (the fallback produced a correct
skill set), but the recurrence count on this lesson should weight the
`marshall-steward` reconciliation fix (Rule, option a) higher — this is now observed
across at least two plans.
