# Negative-config IT passed for the wrong reason - distroless non-root cannot read mktemp 700 dirs; assert the specific validation marker

A Docker integration test that was supposed to prove API Sheriff *rejects* an
invalid configuration passed green for a long time **for the wrong reason**, and
the mistake was invisible until a marker assertion was added.

The test staged the negative config with `mktemp -d`, which creates the directory
with `0700` permissions owned by the host user. The config was then mounted into
the distroless container, whose non-root runtime user could not traverse/read the
`0700` directory. The container therefore failed at startup with **"configuration
file not found"** — not with the **validation-rejection** the test was written to
assert. The test only checked that the container failed to start, so it stayed
green while exercising a completely different failure path; the actual
config-validation logic under test was never reached.

## Rule

Two independent corrections, both required:

1. **Make mounted config readable by the container's runtime user.** Distroless /
   rootless images run as a non-root UID that cannot read `mktemp -d`'s default
   `0700` dir. `chmod` the staging dir and file to be world-readable (e.g. `0755`
   dir / `0644` file), or create them with an explicit readable mode, before
   mounting.

2. **A negative test must assert on the *specific* failure marker, not merely on
   "it failed".** Assert that the container log contains the validation-rejection
   message (the specific reason under test). Asserting only a non-zero exit / "did
   not start" lets an unrelated failure (missing file, port bind, permission) pass
   the test for the wrong reason.

## Prevention

Whenever a test proves a *negative* (rejection, refusal, error), pin it to the
exact diagnostic that proves the intended code path executed — a log-marker
assertion, a specific exit code, or a structured error field. "Failed somehow" is
not evidence that "failed for the reason under test." For container ITs, treat
mount permissions as part of the test contract: the non-root runtime user must be
able to read every mounted path.

## Recurrence (2026-07-21, plan anchor-types-assets)

The same "negative test passes for the wrong reason" class recurred through a
**stale-image** mechanism. The negative-path fixture `verify-invalid-config-fails.sh`
reused whatever `api-sheriff:distroless` docker image tag already existed locally
instead of building the image from the current schema. So the fixture exercised an
**old image built against an old schema** and never validated the current
config-schema rejection at all — it went green locally while proving nothing about
the change under test. CI (building a fresh image) exercised the current schema.

**Broadened rule**: a negative-config test's contract includes *which artifact it
runs against*. A fixture that reuses an existing image tag (or any pre-built artifact)
must either rebuild from the current source or pin/verify the image digest, otherwise
it silently certifies a stale artifact. Combine with the specific-marker assertion
above: assert the current-schema rejection marker AND ensure the image under test was
built from the current schema.

## Impact

Applies to every negative-path Docker IT / fixture in the integration-tests module
that stages fixtures via `mktemp` and mounts them into, or reuses a tag of, the
distroless api-sheriff image. A negative test that passes for the wrong reason —
wrong failure path or stale artifact — is worse than no test: it certifies a
validation path that was never exercised. The stale-image variant was one of three
consecutive CI re-settles on plan anchor-types-assets.
