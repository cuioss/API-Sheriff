# OpenRewrite recipe mutation passes pre-commit but leaves the tree broken for the next build - run a clean full verify after any recipe run

An OpenRewrite recipe (`UseMapOf`) applied during the quality-gate / pre-commit run rewrote production source. That run itself finished **green** — the recipe mutated the tree *after* the compile/test phases it was gated behind had already passed, so `verify -Ppre-commit` reported success. The mutated tree, however, was left in a state that broke the **next** clean build.

The trap: pre-commit incremental verification validates the tree it *started* with, not the tree the recipe *leaves behind*. A recipe that reformats or restructures code (import reordering, `Map.of` substitution, generic re-inference) can produce a diff that compiles against the warm reactor state but fails a cold, from-scratch resolution — and pre-commit never re-checks the post-mutation tree.

## Rule

After any build that applies OpenRewrite recipes (or any other source-mutating build step), do **not** trust the passing pre-commit that produced the mutation. Run a **clean full verify** over the post-mutation tree before committing:

- The passing run that emitted the recipe diff is not evidence the diff is buildable.
- Re-run `verify` (full, not `-Ppre-commit` incremental) so the mutated sources are compiled and tested from a clean state.
- Only commit once the post-mutation tree passes a clean verify — the pre-commit green that produced the change is stale by construction.

## Prevention

Treat source-mutating build steps as producing a new tree that has never been verified. Gate the commit on a verify pass that reads the mutated sources, not on the run that generated them. This is the OpenRewrite-specific instance of the general "the tree you tested is not the tree you're committing" hazard.

## Impact

Applies to every build invocation that runs OpenRewrite recipes across the project (the `rewrite:run` / recipe-bearing profiles). A recipe-induced breakage that passes pre-commit surfaces only on the next developer's or CI's clean build — far from the change that caused it, with no in-run signal at the point of mutation.
