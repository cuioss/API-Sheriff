# Security deliverable claimed a vuln class closed while the control was unimplemented - verify the control exists, do not trust the narrative

A deliverable narrative claimed that a "canonicalize-and-confine" implementation had
**closed the symlink-escape class** of path-traversal vulnerability for asset serving.
The finalize security audit found the requirement was in fact **unimplemented** — the
canonicalization/confinement control the narrative described did not exist in the
production path, so the symlink-escape class was still open. The deliverable's prose
was mistaken for evidence that the control shipped.

## Rule

- A deliverable's claim that a vulnerability class is "closed" / "handled" /
  "mitigated" is NOT evidence the control exists. Independently verify the control in
  the production code path — read the actual canonicalization/confinement call site and
  its enforcement, or exercise the attack (a symlink pointing outside the asset root)
  against the built artifact.
- Security acceptance criteria must be checked against the **implementation**, never
  against the deliverable's self-description. Narrative and implementation drift silently;
  only the code and a live attack close the gap.

## Prevention

For every security requirement with a named vuln class (path traversal, symlink escape,
SSRF, injection), the finalize/audit stage must trace the claimed control to a concrete
enforcement point and, where feasible, run the negative attack against the running
artifact. Treat "the deliverable says it's closed" as an unverified assertion until the
control is located in code.

## Impact

Found by the finalize security audit on this plan — a claimed-closed vuln class was
actually open. Recurs whenever a security deliverable is accepted on its narrative
rather than on verified enforcement in the production path.
