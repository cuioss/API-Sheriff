# CDI lazy proxy defeats @Observes StartupEvent fail-fast validation - the injected validator is never assembled unless a method is actively invoked

The design intent was fail-fast-at-startup: a `@Observes StartupEvent` observer had the validator bean injected so that any invalid configuration would abort application boot. It silently did not work. CDI injects a **lazy client proxy** for the validator — the proxy is created eagerly, but the underlying bean instance (and therefore its `@PostConstruct` assembly / eager validation logic) is only materialized on the **first method invocation** through the proxy. The observer merely *held* the injected reference without calling anything on it, so the validator was never assembled and the fail-fast validation never ran at startup.

The defect slipped through unit tests because a unit test that exercises the validator **calls a method on it**, which forces proxy resolution and assembly — so the validation fires under test but not in the real startup path. It was caught only by the PR review bot (CodeRabbit Critical `0f50e9`).

## Rule

To force eager assembly of an injected CDI bean inside a `StartupEvent` observer, the observer must **actively invoke a method** on the injected reference — injection alone does not trigger `@PostConstruct` through a lazy proxy:

- In the `@Observes StartupEvent` handler, call a real method on the validator (e.g. `validator.validate()` / `validator.assemble()` / a deliberate no-op touch) so the proxy resolves and the bean's assembly + validation logic runs.
- Alternatively, mark the bean for eager initialization (e.g. `@Startup` / an eager scope) so the container assembles it at boot independently of proxy touches.
- Never treat "the bean is injected into the startup observer" as evidence that its startup-time side effects have run.

## Prevention

When a startup-time side effect (fail-fast config validation, resource pre-warm, health assertion) is the whole point of injecting a bean, add a test that asserts the side effect fired via the **startup event path** — not via a direct method call on the bean — so the lazy-proxy gap is visible in-run. A unit test that calls the bean directly will pass even when the startup wiring is inert.

## Impact

Applies to every fail-fast / eager-initialization pattern that relies on CDI injection into a lifecycle observer. A lazy-proxy-defeated startup validator is worse than none: it manufactures the belief that misconfiguration aborts boot while the gateway actually starts with invalid configuration.
