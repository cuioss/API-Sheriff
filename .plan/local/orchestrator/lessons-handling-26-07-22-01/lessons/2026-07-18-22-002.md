# Sonar java:S3655 fires on logically-guarded Optional access - hoist the Optional into a local before isPresent/get

Sonar rule `java:S3655` ("Optional value should only be accessed after calling
isPresent()") fired twice during this plan on accesses that were in fact
logically guarded. Sonar's dataflow engine could not connect the guard to the
access in two recurring shapes:

1. **Repeated accessor calls.** `getFoo()` returns an `Optional`. Guarding with
   `if (getFoo().isPresent())` and then calling `getFoo().get()` in the body reads
   as *two independent Optionals* to the analyzer — the second call is treated as
   unguarded, so S3655 fires.

2. **Derived Optionals.** Guarding one Optional and then accessing a *different*
   Optional derived from it (e.g. via `.map(...)` / a second lookup) breaks the
   present-ness proof chain the analyzer needs.

Both are false positives about program correctness but true positives about
*style*: the code relies on the reader (and re-evaluation) rather than a single
proven-present reference.

## Rule

Hoist the Optional into a local variable once, then guard and consume that same
local — do not call the accessor twice:

```java
// triggers S3655
if (getConfig().isPresent()) {
    use(getConfig().get());
}

// clean: one reference the analyzer can prove present
Optional<Config> config = getConfig();
if (config.isPresent()) {
    use(config.get());
}
// or prefer the guard-free forms:
getConfig().ifPresent(this::use);
Config config = getConfig().orElseThrow();
```

For derived Optionals, resolve to the concrete value at the guard boundary
(`orElse` / `orElseThrow` / `ifPresent`) rather than threading a second Optional
past the check.

## Prevention

Treat S3655 as a signal to remove the double-evaluation, not to suppress. Prefer
`ifPresent` / `map` / `orElseThrow` over the `isPresent()` + `get()` pair; when the
pair is unavoidable, bind the Optional to a single local first. This keeps the
in-run quality gate green and avoids the CI loop-back this plan hit twice.

## Impact

Applies across the api-sheriff module wherever accessor methods return
`Optional`. The fix is a mechanical local-hoist / accessor-consolidation and never
warrants an `// NOSONAR` suppression.
