# Native docker IT tier deferred to CI hides ConfigValidator boot-posture failures - run native IT locally before push

The Docker/native integration-test tier (`verify -Pintegration-tests -pl integration-tests`)
was not run locally during this plan; those runs were deferred to CI. As a result,
two independent `ConfigValidator` boot-posture failures surfaced only remotely, each
costing a CI loop-back round:

1. A config fixture was missing the endpoint-level `require:none` auth posture, so the
   application aborted boot on the endpoint-posture validation rule.
2. A gRPC route fixture violated the anchor-namespace containment rule, so boot aborted
   on the gRPC anchor-namespace validation rule.

Both are startup-time `ConfigValidator` rejections that only manifest when the native
image actually boots against the fixture inside the docker-compose IT stack. The JVM
unit suite and the local quality gate never boot the assembled config against the
validator in container context, so neither failure produced any in-run signal — they
were invisible until CI built the image and ran the IT suite.

## Rule

- The docker/native integration-test tier is a distinct gate that a JVM-green local
  build does NOT cover. When a change adds or edits config fixtures (route/endpoint/anchor
  YAML, auth posture, gRPC namespace), run the native IT suite locally
  (`verify -Pintegration-tests -pl integration-tests -am`, after building the native image)
  before pushing — do not defer the first native-IT execution to CI.
- Treat every `ConfigValidator` posture/containment rule (endpoint-level `require:none`,
  gRPC anchor-namespace containment, and their siblings) as boot-time-only: a fixture that
  violates one aborts application startup and is caught only by an actual container boot,
  never by unit tests or the compile/test quality gate.

## Prevention

Make a local native-IT run part of the pre-push checklist for any config-fixture change.
Use the `run-integration-tests` skill's split-step diagnosis loop (build native image →
bring up the stack → read container boot logs) to surface `ConfigValidator` boot failures
locally, so posture/containment defects fail in-run instead of after push. If a local
native-IT run is genuinely infeasible, treat the FIRST CI round as the real gate and
budget for the loop-back rather than assuming local green certifies the IT tier.

## Impact

Deferring the docker/native IT tier to CI cost two consecutive CI loop-back rounds on this
plan (PR #88), each a full image-build + IT-suite cycle. This is a distinct root cause from
the native-reflection registration gap (lesson 2026-07-21-04-001) and the warm-daemon /
stale-artifact masking (lesson 2026-07-19-17-001): here the entire IT tier was simply never
run locally, so nothing masked the defect — it was just never looked for until CI. Recurs on
any config-fixture change validated only through JVM-mode local gates.
