# Content-Length is an end-to-end header not hop-by-hop; curl HTTP/2 vs RestAssured HTTP/1.1 masks the bug locally

The gateway's ResponseStage treated the end-to-end `Content-Length` header as if it
were hop-by-hop and stripped it, which dropped streamed response bodies over HTTP/1.1.
The defect was invisible in local smoke tests because `curl` negotiates HTTP/2 by
default, while the RestAssured-based automated tests speak HTTP/1.1 — so the bug only
manifested on the HTTP/1.1 path that local manual testing never exercised.

## Root cause

Per RFC 7230, only a fixed set of headers are hop-by-hop (Connection,
Keep-Alive, Proxy-Authenticate, Proxy-Authorization, TE, Trailer,
Transfer-Encoding, Upgrade, plus anything listed in `Connection`). `Content-Length`
is end-to-end and MUST be forwarded. A proxy/gateway that strips it corrupts framing
and can drop the body depending on the protocol version.

## Rule

- Never strip `Content-Length` (or any non-listed header) as if it were hop-by-hop.
  Maintain the RFC 7230 hop-by-hop allowlist explicitly; forward everything else.
- Protocol-version divergence between manual and automated clients hides
  version-specific bugs. `curl` defaults to HTTP/2 against TLS; RestAssured and many
  Java HTTP clients default to HTTP/1.1. When smoke-testing a gateway/proxy manually,
  force the same version the automated suite uses (`curl --http1.1`) — and test BOTH
  HTTP/1.1 and HTTP/2 explicitly for any header/framing-sensitive behavior.

## Impact

Streamed bodies silently dropped on HTTP/1.1 while every local curl check passed.
Recurs on any header-forwarding or framing change validated only via default-HTTP/2 curl.
