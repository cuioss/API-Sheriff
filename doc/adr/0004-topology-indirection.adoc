= ADR-0004 -- Topology Indirection: Single-URL Alias with Env Override
:toc:
:toclevels: 2

== Status

Accepted; the *override channel* is superseded by link:#_amendment_a1[Amendment A1] below
(in-file `${VAR:-default}` placeholder instead of the convention-named env var). The
single-URL-per-alias model is unchanged.

== Context

Endpoint (route) configuration should be *portable* -- movable between dev, staging, and production
without edits -- while the concrete backend locations differ per environment. The chosen structure
(see link:../configuration.adoc[Configuration Model]) abstracts every backend host behind an alias
(`APP1`, `KEYCLOAK`, ...) referenced by the endpoint files, and binds each alias to a real location
in a separate `topology.properties`, overridable by an environment variable
(`TOPOLOGY_<ALIAS>=https://...`).

One design question remained: should a topology entry be a *single URL*
(`https://app1.example.com:8443/ctx`) or *split components* (scheme / host / port / path)? We
researched what the majority of widely-used systems do for human-authored, env-overridden config.

=== Research findings

* *No landslide, but a clean axis.* Machine-generated / service-discovery data planes (Envoy xDS,
  Kubernetes, APISIX admin API, HAProxy runtime) favour *split components* -- easy to populate and
  diff programmatically. Human-authored declarative surfaces (Traefik `url`, Spring Cloud Gateway
  `uri`, Nginx `proxy_pass`, Kong's `url` shorthand) favour a *single URL* -- the compact,
  copy-pasteable form. API Sheriff's topology is the *human-authored, env-overridden* surface, so
  the single-URL camp applies.
* *12-factor precedent.* Config treats a backing service's location as *one atomic resource handle*
  (`DATABASE_URL`, `REDIS_URL`, ...), not a family of `_HOST`/`_PORT`/`_SCHEME` vars. This is the
  strongest precedent and points at one URL per alias.
* *Kong's hybrid.* Kong stores `protocol`/`host`/`port`/`path` as separate fields but accepts a
  write-only `url` that is *decomposed on write*. URLs win for input ergonomics; components win for
  validation/partial-update -- so accept the URL at the boundary and decompose once.
* *Trade-offs.* One env var beats four coordinated ones (atomic change, no half-applied override).
  A single URL validates in one shot (`java.net.URI`) and then yields the components anyway;
  assembling a URL from parts is strictly more error-prone (default ports, IPv6 brackets, trailing
  slashes). The only real argument for splitting is partial override (change only the port), which
  is rare and cheap to work around.

== Decision

* *A topology entry is a single URL per alias.* `topology.properties`:
  `APP1=https://app1.internal:8443/app1`.
* *Environment variable `TOPOLOGY_<ALIAS>` overrides the file and takes precedence.*
  Precedence, highest first: `TOPOLOGY_<ALIAS>` env var -> `topology.properties`.
  *(Superseded by link:#_amendment_a1[Amendment A1]: the same precedence, expressed as a
  visible in-file placeholder instead of a convention-named lookup.)*
* *Decompose-on-read (Kong model).* On load, each resolved URL is validated and decomposed once into
  `{scheme, host, port, base-path}`; the rest of the gateway works with the structured components. A
  declared `route.upstream.path` *replaces* the alias-derived base path (it does not append to it),
  and the forward URI is `stripTrailingSlash(effective base path) + <request-path remainder after
  the matched prefix>`, where the effective base path is the `route.upstream.path` when declared and
  the alias-derived base path otherwise (see link:../configuration.adoc[Configuration -- `upstream`]).
* *No component split initially.* Per-component overrides (e.g. `TOPOLOGY_APP1_PORT`) are *not*
  implemented now. They may be added later as a narrow escape hatch (mirroring Kong: full URL as the
  ergonomic default, components as fine-grained override) only if a real partial-override need appears.
* *Endpoint files never contain a concrete host* -- only the alias and an alias-relative path. Moving
  a deployment changes only the topology (file or env), never the endpoint files.

== Consequences

* Relocating a backend is a one-line change (`TOPOLOGY_APP1=...`), with no edit to portable endpoint
  files -- the intended "move a deployment by editing topology only" property.
* Matches the dominant human-authored idiom (12-factor, Traefik, Spring, Nginx, Kong shorthand),
  minimizing surprise and parsing risk.
* Validation is single-shot and fail-fast at startup (an unresolved alias or malformed URL fails the
  boot).
* Partial override (port only) requires restating the whole URL until/unless the component escape
  hatch is added -- an accepted, low-cost trade.

[[_amendment_a1]]
== Amendment A1 (2026-07-15) -- Override channel becomes an in-file placeholder

The *single-URL-per-alias* decision above is unchanged. What A1 supersedes is the *override
channel*: the convention-named environment variable (`TOPOLOGY_<ALIAS>` wins over the file),
which was implemented as an invisible Java-class-level lookup a reader cannot discover from
the artifact. It is replaced by the unified in-file substitution mechanism decided with the
anchors work (link:../plan/03-endpoint-anchors.adoc[Plan 03], D4): every override point is a
visible shell-style placeholder in the artifact it affects --

[source,properties]
----
APP1=${TOPOLOGY_APP1:-https://app1.internal:8443/app1}
----

`${VAR}` is a required substitution (unset fails the boot); `${VAR:-default}` expresses "file
value as default, environment wins" -- the role the convention override played. The familiar
`TOPOLOGY_<ALIAS>` variable *names* survive as a sample-config convention; the *mechanism* is
the placeholder, resolved once before schema validation. Everything else in this ADR
(one URL per alias, decompose-on-read, no component split, no concrete host in endpoint
files) stands. Implementation lands with Plan 03; `configuration.adoc`'s topology section is
rewritten in the same PR.
