== Architecture

* *Proxy model:* Native Go reverse proxy -- built from scratch on Go's HTTP server, no external proxy dependency
* *Core components:* Gateway (traffic proxy), Dashboard (UI/management -- commercial), Pump (metrics export)
* *Middleware chain:* Extensible plugin architecture; middleware pipeline processes requests through configurable stages
* *Plugin system:* Rich plugins via Go, Python, JavaScript, Lua, or any gRPC-capable language
* *API definition:* JSON-based; supports Tyk OAS (OpenAPI compliant) and Tyk Classic formats
* *GraphQL:* Native engine (not just proxying -- includes schema stitching and federation)

IMPORTANT: In OSS mode, multiple gateway replicas do *not* sync API definitions automatically.
The docs explicitly warn against running `ReplicaCount > 1` for OSS gateways without the Dashboard/control plane.

== Stateless Design

*Stateful at the system level.* Redis is mandatory for rate limiting, token/session storage, and distributed coordination.
Individual gateway nodes are stateless proxies, but the system cannot operate without Redis.

== Token Mediation / BFF Support

Evaluated against https://www.ietf.org/archive/id/draft-ietf-oauth-browser-based-apps-13.html[IETF draft-ietf-oauth-browser-based-apps]:

[cols="1,1,2"]
|===
| Variant | Supported? | Notes

| Token-Mediating Backend (encrypted cookie)
| No
| The gateway does not act as an OIDC relying party and does not encrypt tokens into cookies.

| BFF Proxy (server-side session)
| No
| The gateway does not manage token sessions for browser-based SPAs.
|===

The gateway does *not* act as an OIDC Relying Party -- it does not register with the IdP, does not redirect browsers to the IdP, and does not exchange authorization codes for tokens.

The *Tyk Identity Broker (TIB)* is a separate component that can initiate OAuth authorization code flows with external IdPs.
However, TIB brokers the initial authentication and hands off a Tyk-issued token to the client -- it does *not* encrypt the bearer token into an HttpOnly cookie or perform cookie-to-bearer mediation on subsequent requests.

Since Tyk 5.7.0, the dedicated OIDC auth option in the gateway was deprecated in favor of pure JWT validation, moving further away from acting as a confidential client.

=== Automatic Token Refresh

*Not supported.* The gateway validates tokens but expects the client to handle refresh token flows.
Tyk supports the `refresh_token` grant type as an OAuth2 *server*, but the client must explicitly call it.

=== Step-Up Authentication (RFC 9470)

*Not supported.* The gateway does not parse upstream `WWW-Authenticate` headers for `insufficient_user_authentication`, `acr_values`, `max_age`, or `scope` challenges.
It cannot trigger re-authorization with elevated authentication levels or additional scopes based on upstream responses.

== Forwarded Header / RFC 7239

*No evidence of RFC 7239 support.* Documentation only references `X-Forwarded-For` with `xff_depth` configuration for anti-spoofing.
The standardized `Forwarded` header is not mentioned.

== Strengths

* High performance (built in Go, no external proxy dependency)
* Full-featured OSS gateway (auth, rate limiting, transforms, GraphQL all included)
* Multi-protocol (REST, GraphQL native engine, gRPC, WebSocket, TCP, SOAP)
* Flexible plugin system (Go, Python, JS, Lua, gRPC)
* Good documentation (developer-first)
* Kubernetes-native (Tyk Operator with CRDs)
* Multi-cloud / no vendor lock-in

== Weaknesses

* Operational complexity (Gateway + Redis mandatory; Dashboard + MongoDB/PostgreSQL for enterprise)
* OSS scaling limitation (multiple nodes don't sync without commercial Dashboard)
* Smaller community than Kong (~10.7k vs ~42.9k stars)
* No WAF
* No HTTP/3
* No RFC 7239 support
* Dashboard requires license for commercial use

The commercial lock is on management, multi-node orchestration, and developer portal -- not on gateway traffic-processing features.
