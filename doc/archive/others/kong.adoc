== Architecture

* *Proxy model:* Reverse proxy built on Nginx/OpenResty
* *Plugin architecture:* Request lifecycle hooks -- plugins inject logic at various phases (access, header_filter, body_filter, log, etc.)
* *Control Plane / Data Plane separation:* Supported since Kong 2.x; CP pushes configuration, DP handles traffic
* *Plugin execution:* Plugins run in-process via LuaJIT (high performance, low overhead)

== Stateless Design

*Primarily stateless.* Data plane nodes receive configuration from the control plane or declarative config file and keep state in memory.
They can be horizontally scaled without shared state.

In DB-less mode, the entire deployment is stateless.
Rate limiting counters may require shared storage (Redis or database) for cluster-wide consistency.

== Token Mediation / BFF Support

Evaluated against https://www.ietf.org/archive/id/draft-ietf-oauth-browser-based-apps-13.html[IETF draft-ietf-oauth-browser-based-apps]:

[cols="1,1,2"]
|===
| Variant | Supported? | Notes

| Token-Mediating Backend (encrypted cookie)
| *Yes* (*Enterprise only*)
| Kong's OIDC plugin (via lua-resty-openidc / lua-resty-session) supports this as the *default* mode.

| BFF Proxy (server-side session)
| *Yes* (*Enterprise only*)
| By switching `session_storage` to `redis` or `memcache`, tokens are stored server-side.
|===

Kong's OIDC plugin (*Enterprise only*) makes Kong a full *OIDC Relying Party (confidential client)*:

. Kong is registered with the IdP using `config.client_id` and `config.client_secret` (required configuration).
. When an unauthenticated request arrives, Kong redirects the browser (HTTP 302) to the IdP's authorization endpoint, including PKCE `code_challenge`.
. The user authenticates at the IdP.
. The IdP redirects back to Kong at the configured `redirect_uri`.
. Kong exchanges the authorization code for tokens by calling the IdP's token endpoint *server-to-server*, authenticating with its client credentials.
. Kong validates the tokens (cryptographic signature, expiry).
. Tokens are encrypted (AES-256-GCM via lua-resty-session) into an `HttpOnly` session cookie.
. On subsequent requests, Kong decrypts the cookie and forwards the access token upstream as `Authorization: Bearer`.

Underlying library: lua-resty-session (same foundation as APISIX).
Configuration: `config.session_storage = "cookie"` (default), `config.auth_methods = ["authorization_code", "session"]`.

IMPORTANT: This capability is *Enterprise only*. The OSS edition (frozen at 3.9.1) does not include the OIDC plugin.
The community plugin `nokia/kong-oidc` (also based on `lua-resty-openidc`) provides similar functionality for Kong OSS, but is not officially supported.

=== Automatic Token Refresh

*Yes* (*Enterprise only*). The `refresh_tokens` parameter (default: `true`) enables the plugin to refresh "(soon to be) expired access tokens if the plugin has a refresh_token available."

* Transparent to the client in session mode (no redirect during normal requests).
* Refresh token is stored in the encrypted session cookie alongside the access token.
* If the refresh token has expired, the session is destroyed (`unauthorized_destroy_session = true` by default) and the client must re-authenticate.

=== Step-Up Authentication (RFC 9470)

*Not supported.* The OIDC plugin does not parse upstream `WWW-Authenticate` headers for `insufficient_user_authentication`, `acr_values`, `max_age`, or `scope` challenges.
It cannot trigger re-authorization with elevated authentication levels or additional scopes based on upstream responses.
There are no open issues or feature requests for RFC 9470 support.

== Forwarded Header / RFC 7239

*RFC 7239 (`Forwarded` header): Not supported.* This is a known limitation tracked since 2015 (https://github.com/Kong/kong/issues/270[GitHub issue #270]).
The underlying Nginx `ngx_http_realip_module` does not support RFC 7239.

Legacy `X-Forwarded-*` headers (`X-Forwarded-For`, `-Proto`, `-Host`, `-Port`, `-Path`) are fully supported from trusted sources.

== Strengths

* Extremely high performance (Nginx/LuaJIT foundation, sub-millisecond plugin overhead)
* Massive plugin ecosystem (200+ plugins in Kong Hub)
* Kubernetes-native with Gateway API conformance
* Flexible deployment (DB-less, hybrid, traditional)
* Strong community and wide adoption (~42.9k stars)
* Mature project (since 2015)
* Control Plane / Data Plane separation

== Weaknesses

* *Open-core model tightening:* OSS edition frozen at 3.9.1; 3.10+ requires Enterprise license
* No RFC 7239 support
* No HTTP/3 support
* No native request aggregation (limits BFF use cases)
* Many production-critical features are Enterprise-only (OIDC, vault, RBAC, advanced rate limiting, WAF)
* Memory consumption can be high with many plugins or large configurations
* Enterprise pricing criticized as expensive
