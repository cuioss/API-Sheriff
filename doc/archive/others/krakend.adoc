== Architecture

* *Type:* Layer 7 API Gateway, reverse proxy, and API aggregator
* *Design:* Stateless, declarative, immutable infrastructure
* *No database required:* Operates without any external datastore; uses ~100 MB RAM under high traffic
* *Cluster mode:* Every node operates independently; no coordination or centralized persistence needed
* *Scaling:* True linear horizontal scalability with no single point of failure
* *Proxy model:* Sits between clients and backends; aggregates, transforms, and filters responses from multiple microservices

== Stateless Design

*Stateless.* This is a core architectural principle.
No database, no coordination between nodes, no centralized state.
Each node is self-contained with its configuration file.

Exception: Enterprise Edition offers Redis-backed stateful rate limiting for cross-node coordination.

== Token Mediation / BFF Support

Evaluated against https://www.ietf.org/archive/id/draft-ietf-oauth-browser-based-apps-13.html[IETF draft-ietf-oauth-browser-based-apps]:

[cols="1,1,2"]
|===
| Variant | Supported? | Notes

| Token-Mediating Backend (encrypted cookie)
| No
| KrakenD does not initiate OAuth authorization code flows. It validates tokens only.

| BFF Proxy (server-side session)
| No
| KrakenD does not manage token sessions.
|===

The gateway does *not* act as an OIDC Relying Party -- it does not register with the IdP as a confidential client, does not redirect browsers to the IdP, and does not exchange authorization codes for tokens.

KrakenD supports JWT *validation* (including reading JWTs from cookies via `cookie_key`) and the OAuth *client credentials* grant (machine-to-machine), but it does *not* support the authorization code flow.
It cannot redirect browsers to an IdP, handle callbacks, or exchange authorization codes.
This is confirmed by https://github.com/krakend/krakend-ce/issues/348[GitHub issue #348] and https://github.com/krakend/krakend-ce/issues/459[#459].

NOTE: KrakenD's "BFF" positioning refers to *API aggregation* (merging responses from multiple backends), not OAuth token mediation.

=== Automatic Token Refresh

*Not supported* for user tokens. KrakenD never holds user refresh tokens since it does not support the authorization code flow.
For machine-to-machine *client credentials* tokens, KrakenD automatically refreshes ~10 seconds before expiry.

=== Step-Up Authentication (RFC 9470)

*Not supported.* KrakenD does not parse upstream `WWW-Authenticate` headers for `insufficient_user_authentication`, `acr_values`, `max_age`, or `scope` challenges.
It validates JWT claims/scopes but has no mechanism to react to upstream challenges by triggering re-authorization.

== Strengths

* Extreme performance (Go, stateless, ~100 MB RAM)
* True stateless architecture (no database dependency, linear horizontal scaling)
* Strong BFF/aggregation (purpose-built for aggregating microservice responses)
* Declarative GitOps model (configuration-as-code, no programming required)
* Linux Foundation backing (core Lura framework under open governance)
* Comprehensive observability (native support for all major telemetry stacks)
* Zero-trust security model (headers and parameters not forwarded by default)
* Predictable Enterprise pricing (not tied to API count or throughput)

== Weaknesses

* *Open-core limitations:* WebSocket, gRPC client/server, SOAP, SSE, wildcard routes, URL rewrite, API keys, and basic auth are all Enterprise-only
* No HTTP/3 support
* No RFC 7239 support
* Static configuration only (no runtime dynamic configuration without restart)
* Smaller community (~2.6k stars vs Kong ~42.9k or APISIX ~16.3k)
* No admin UI (KrakenD Designer is a config tool, not a runtime admin dashboard)
* Stateful rate limiting is Enterprise-only (CE rate limiting is per-node only)
* Configuration files can become verbose with duplication
