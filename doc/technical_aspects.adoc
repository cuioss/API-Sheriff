== Overview

This document describes the CUI libraries and frameworks used by API Sheriff, covering HTTP client communication, security validation, OAuth token handling, logging, and testing.

== Downstream API Communication

Library: **cui-http**

=== HttpAdapter & HttpHandler

`HttpAdapter` provides an async-first API for HTTP methods (`get()`, `post()`, `put()`, `delete()`) backed by `CompletableFuture`.
`HttpHandler` is a builder-based HTTP client wrapper with automatic SSL context creation, thread-safe after construction.

Composable adapters add cross-cutting concerns:

* `ResilientHttpAdapter` -- non-blocking retry with exponential backoff
* `ETagAwareHttpAdapter` -- HTTP caching via ETag / 304 Not Modified

=== HttpResult Pattern

`HttpResult<T>` is a sealed interface with two record types:

* `HttpResult.Success<T>` -- carries content, ETag, and HTTP status
* `HttpResult.Failure<T>` -- carries error message, cause, fallback content, `HttpErrorCategory`, ETag, and status

`HttpErrorCategory` classifies errors: `NETWORK_ERROR`, `SERVER_ERROR`, `CLIENT_ERROR`, `INVALID_CONTENT`, `CONFIGURATION_ERROR`.

Usage follows exhaustive pattern matching (switch expressions) or simple `isSuccess()` checks.
The `map()` method transforms content while preserving metadata.
Failures may carry fallback content for graceful degradation with cached data.

NOTE: `HttpResult<T>` *materializes the whole body* into the in-memory `T`. That is exactly
right for the *control plane* -- the small, bounded JSON calls the gateway makes itself (OIDC
discovery, JWKS, token exchange, revocation) -- but it is *not* used for the *data plane*:
proxied client bodies (large responses, downloads, SSE, chunked/long-lived streams) are
*streamed* with backpressure and never buffered into an `HttpResult<byte[]>`/`HttpResult<String>`.
The boundary, and the competitor evidence behind it, is
link:adr/0006-body-streaming-vs-httpresult.adoc[ADR-0006].

== HTTP Security Validation

Library: **cui-http** (security module)

=== Validation Framework

`HttpSecurityValidator` is the functional interface for input validation:
`Optional<String> validate(@Nullable String value) throws UrlSecurityException` -- a violation
throws, a `null` input yields `Optional.empty()`.

Dedicated validation pipelines:

[cols="1,2"]
|===
| Pipeline | Scope

| `URLPathValidationPipeline`
| Path traversal, directory traversal patterns, CVE exploits

| `HTTPHeaderValidationPipeline`
| Header injection, CRLF attacks

| `URLParameterValidationPipeline`
| Query-parameter values (request-*body*/form validation was removed from `cui-http` --
  `PipelineFactory` rejects `BODY`; body limits are the application's, i.e. the gateway's, job)
|===

The pipelines compose a shared stage toolbox -- `LengthValidationStage`,
`CharacterValidationStage`, `PatternMatchingStage`, `DecodingStage` (URL percent-decoding,
overlong-UTF-8 detection), `NormalizationStage` (Unicode), `AllowBlockListStage` -- but each
pipeline orders them differently, and the order is deliberate:

* *Path*: length -> character -> pattern (*pre-decode* -- patterns must run before
  normalization can mask them) -> decode -> normalize -> pattern (again, post-decode).
* *Parameter*: length -> character -> decode -> normalize -> pattern.
* *Header*: length -> character -> normalize -> pattern -> allow/block list -- *no decoding
  stage at all* (header values are not percent-encoded).

Configuration via `SecurityConfiguration` with sensible defaults (path length 4096, parameter count 100, header count 50, cookie count 20, body size 5 MB).

=== Forwarded-Header Resolution

The `de.cuioss.http.forwarded` package (specified in
`cui-http/doc/forwarded-header-resolution.adoc`) resolves the reverse-proxy header family into
one normalized, sanitized, immutable structure -- API Sheriff's
link:configuration.adoc#_forwarded_headers[`forwarded` policy] is a thin configuration layer
over it:

* `ForwardedHeaderResolver` -- entry point; immutable, thread-safe, transport-agnostic
  (consumes a `Function<String,String>` header accessor). `resolve(...)` never throws --
  pipeline sanitization failures increment the `SecurityEventCounter`, the raw-value guards
  log `HTTP-120`..`HTTP-123` WARNs, and plain trust misses are silently not honored.
* `ResolvedForwarding` -- immutable result record: `scheme`, `host`, `port`, `contextPath`,
  `clientIp`, each present only when honored; `empty()` is the secure default.
* `ForwardedResolverConfig` -- trust model: `trustedProxies` (CIDR, drives the right-to-left
  client-IP chain walk), `trustAll` (honor sanitized scheme/host/port/prefix),
  `allowedContextPaths`, and a `securityConfig` hook reusing the `SecurityConfiguration`
  presets for header-value sanitization.
* Serialization: `toXForwardedHeaders()` (`X-Forwarded-Proto/-Host/-Port/-Prefix/-For`) and
  `toForwardedHeader()` (one strict RFC 7239 value, `for=..;host=..;proto=..`, no `by=`) --
  regeneration from resolved facts, never propagation of inbound chains.

=== Attack Pattern Test Databases

The test artifact provides curated real-world attack pattern databases for parameterized testing:

* **CVE-based**: `ApacheCVEAttackDatabase`, `IISCVEAttackDatabase`, `NginxCVEAttackDatabase`
* **Standards-based**: `OWASPTop10AttackDatabase`, `ModSecurityCRSAttackDatabase`, `OWASPZAPAttackDatabase`
* **Specialized**: `HomographAttackDatabase`, `IPv6AttackDatabase`, `ProtocolHandlerAttackDatabase`, `IDNAttackDatabase`
* **Legitimate patterns**: `LegitimatePathPatternsDatabase`, `EdgeCaseValidURLsDatabase` -- ensuring valid inputs pass

Each database provides `AttackTestCase` records with attack string, expected failure type, description, and rationale.

== OAuth Token Handling

Library: **token-sheriff-validation** (`de.cuioss.sheriff.token:token-sheriff-validation`; Quarkus
integration via `token-sheriff-validation-quarkus`)

=== Purpose

Comprehensive JWT validation for multi-issuer OAuth/OpenID Connect environments with a focus on offline validation.
Tokens are validated entirely offline using periodically fetched JWKS keys -- no synchronous calls to identity providers required.

=== Core Components

[cols="1,2"]
|===
| Component | Description

| `TokenValidator`
| Main facade orchestrating validation pipelines with caching and metrics. Pre-creates per-issuer validators at construction time.

| `AccessTokenContent`
| Type-safe access token representation with `getScopes()`, `getRoles()`, `getGroups()`, `providesScopes()`, and missing-scope reporting.

| `IdTokenContent`
| OIDC ID token for authentication flows.

| `UnvalidatedRefreshToken`
| Refresh tokens are treated as opaque: `createRefreshToken(...)` performs minimal checks and
  returns them unvalidated (there is deliberately no validated refresh-token content type).

| `IssuerConfig`
| Per-identity-provider configuration: issuer, audience, JWKS source, DPoP settings, signature algorithms, custom claim mappers.

| `IssuerConfigCache`
| Thread-safe resolver managing multiple issuer configurations (a class named
  `IssuerConfigResolver` exists only in the Quarkus module, where it resolves MicroProfile
  config properties -- a different concern).
|===

=== Validation Pipeline

Layered fail-fast approach:

. `TokenStringValidator` -- null, blank, size checks
. Token parsing (DSL-JSON, not reflection-based) and issuer extraction
. *Cache lookup -- early*, right after minimal parsing and before any further work
. On cache miss: issuer-config resolution via `IssuerConfigCache`
. `TokenHeaderValidator` -- algorithm and key ID verification
. `TokenSignatureValidator` -- JWS signature against JWKS keys
. `TokenClaimValidator` -- issuer, audience, expiration, not-before

Additional capabilities:

* `DpopProofValidator` -- sender-constrained token validation with replay protection
* `AccessTokenCache` -- high-performance cache with configurable eviction
* `SecurityEventCounter` -- thread-safe tracking of suspicious patterns

=== Confidential-Client Engine

Library: **token-sheriff-client** (`de.cuioss.sheriff.token:token-sheriff-client`; Quarkus
integration incl. RFC 9457 exception mapping via `token-sheriff-client-quarkus`)

The OIDC/OAuth confidential-client engine, purpose-built for API Sheriff and used by the two
BFF variants: authorization-code + PKCE retrieval, `state`/`nonce`/mix-up defence, PAR,
sender-constraint, step-up, RP-initiated logout triggering, and the server-side token
lifecycle (refresh with rotation and reuse revocation, RFC 7009 revocation). It wires
`token-sheriff-validation` internally to validate every retrieved token. The authoritative
API and requirement surface is `TokenSheriff/doc/client` (`CLIENT-1..22`); the split between
engine and BFF responsibilities is stated in
link:architecture.adoc#_authentication_layer[Architecture -- Authentication Layer].

== Logging

=== Production Logging

Library: **cui-java-tools** (`de.cuioss.tools.logging`)

`CuiLogger` wraps JUL with structured logging support.

[source,java]
----
private static final CuiLogger LOGGER = new CuiLogger(MyClass.class);
----

Structured messages use `LogRecord` / `LogRecordModel`:

[source,java]
----
private static final LogRecord TOKEN_VALIDATED =
    LogRecordModel.builder()
        .prefix("AUTH")
        .identifier(1)
        .template("Token validated for user %s")   // builder field is template, not message
        .build();

LOGGER.info(TOKEN_VALIDATED.format(username));
// Output: AUTH-1: Token validated for user john.doe   (identifier is not zero-padded)
----

Conventions:

* Always `%s` for substitution
* Exception parameter always first: `LOGGER.error(exception, "Failed for %s", id)`
* Identifier ranges: INFO 1-99, WARN 100-199, ERROR 200-299 -- an *allocation* convention;
  the rendered output is not zero-padded (`AUTH-1`, not `AUTH-001`)
* Lazy evaluation via `Supplier<String>` for expensive debug messages

=== Log Testing

Library: **cui-test-juli-logger**

[source,java]
----
@EnableTestLogger(rootLevel = TestLogLevel.DEBUG)
class MyServiceTest {

    @Test
    void shouldLogOnSuccess() {
        // ... execute ...
        LogAsserts.assertSingleLogMessagePresent(
            TestLogLevel.INFO, "AUTH-1: Token validated for user john.doe");
    }

    @Test
    void shouldLogOnFailure() {
        // ... execute ...
        LogAsserts.assertLogMessagePresentContaining(
            TestLogLevel.ERROR, "Failed", IOException.class);
    }
}
----

Key components:

* `@EnableTestLogger` -- JUnit 5 annotation configuring log capture with per-logger level control
* `TestLogHandler` -- captures `LogRecord` instances in a thread-safe list
* `LogAsserts` -- fluent assertions: presence, absence, singularity, substring matching, throwable filtering

== HTTP Unit Testing

Library: **cui-test-mockwebserver-junit5**

JUnit 5 extension wrapping OkHttp's MockWebServer for testing HTTP/HTTPS interactions.

[source,java]
----
@EnableMockWebServer
class DownstreamClientTest {

    @Test
    void shouldHandleSuccessResponse(MockWebServer server, URIBuilder uriBuilder) {
        server.enqueue(new MockResponse.Builder().body("{\"status\":\"ok\"}").build());
        // OkHttp 5.x mockwebserver3: MockResponse is immutable, built via Builder
        var uri = uriBuilder.addPathSegment("api").addPathSegment("data").build();
        // call downstream client with uri, assert HttpResult
    }
}
----

Key components:

* `@EnableMockWebServer` -- activates extension; options for `useHttps` and `manualStart`
* `URIBuilder` -- fluent builder for test request URIs (path segments, query parameters)
* `MockWebServer` and `SSLContext` -- auto-injected into test method parameters
* `ModuleDispatcherElement` -- interface for modular request dispatching with per-HTTP-method handlers
* `BaseAllAcceptDispatcher` -- default positive responses (GET: 200, POST: 201, PUT/DELETE: 204)
* `@MockResponseConfig` -- declarative response setup via annotation (path, method, status, content)

== Integration Testing

Framework: **RestAssured** (already in place)

[source,bash]
----
./mvnw clean verify -Pintegration-tests -pl integration-tests -am
----
