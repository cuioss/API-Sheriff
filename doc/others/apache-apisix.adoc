== Architecture

* *Proxy model:* Reverse proxy based on Nginx's network layer; routing done entirely in Lua using a radix-tree (implemented in C for performance)
* *Control plane / Data plane split:* Supports combined and separated (decoupled) deployment
* *Dynamic configuration:* Data plane nodes watch etcd for changes and apply them within milliseconds
* *Plugin execution:* Plugins run in defined phases (rewrite, access, balancer, header_filter, body_filter, log); deterministic order by priority number
* *Performance:* Claims 140,000+ QPS single-instance; single-core QPS of 18k with average latency under 0.2ms (project benchmarks, not independently verified)

== Stateless Design

*Stateless data plane.* APISIX nodes hold no persistent state; all configuration is sourced from etcd (or file/API in standalone mode).
This enables horizontal scaling and container orchestration.

== Token Mediation / BFF Support

Evaluated against https://www.ietf.org/archive/id/draft-ietf-oauth-browser-based-apps-13.html[IETF draft-ietf-oauth-browser-based-apps]:

[cols="1,1,2"]
|===
| Variant | Supported? | Notes

| Token-Mediating Backend (encrypted cookie)
| *Yes* (open-source)
| The `openid-connect` plugin supports this as the *default* mode (`session.storage = "cookie"`).

| BFF Proxy (server-side session)
| *Yes* (open-source)
| By switching `session.storage` to `"redis"`, tokens are stored server-side.
|===

The `openid-connect` plugin makes APISIX a full *OIDC Relying Party (confidential client)*:

. APISIX is registered with the IdP using `client_id` and `client_secret` (required configuration).
. When an unauthenticated request arrives, APISIX redirects the browser (HTTP 302) to the IdP's authorization endpoint (discovered via the `discovery` URL).
. The user authenticates at the IdP.
. The IdP redirects back to APISIX at the callback URL (`<original_uri>/.apisix/redirect`).
. APISIX exchanges the authorization code for tokens by calling the IdP's token endpoint *server-to-server*, authenticating with its client credentials (method configurable via `token_endpoint_auth_method`).
. Tokens are encrypted (AES-256-GCM via lua-resty-session) into an `HttpOnly` session cookie.
. On subsequent requests, APISIX decrypts the cookie and forwards the access token upstream via `X-Access-Token` header (or `Authorization: Bearer` when `access_token_in_authorization_header = true`).

Underlying library: `lua-resty-openidc` + `lua-resty-session`.
Session contents are configurable via `session_contents` (access_token, id_token, user claims).

NOTE: This is available in the *open-source edition* (Apache 2.0), making APISIX the only evaluated gateway offering stateless token mediation as a full OIDC Relying Party without a commercial license.

=== Automatic Token Refresh

*Yes.* The `renew_access_token_on_expiry` parameter (default: `true`) enables silent renewal of the access token when it expires, provided a refresh token is stored in the session.

* Transparent to the client (no redirect during normal requests).
* `access_token_expires_leeway` (default: `0` seconds) allows triggering renewal a configurable number of seconds *before* actual expiry.
* Refresh token is stored alongside the access token in the encrypted session cookie.
* If the refresh token itself has expired, the user is redirected to the IdP for full re-authentication.

NOTE: Some users have reported issues with `renew_access_token_on_expiry` in `bearer_only` mode (https://github.com/apache/apisix/discussions/12018[GitHub #12018]).

=== Step-Up Authentication (RFC 9470)

*Not supported.* The `openid-connect` plugin does not parse upstream `WWW-Authenticate` headers for `insufficient_user_authentication`, `acr_values`, `max_age`, or `scope` challenges.
It cannot trigger re-authorization with elevated authentication levels or additional scopes based on upstream responses.
There are no open issues or feature requests for RFC 9470 support.

== Forwarded Header / RFC 7239

*No native RFC 7239 support.* APISIX handles legacy `X-Forwarded-For`, `X-Forwarded-Proto`, `X-Forwarded-Host`, and `X-Forwarded-Port` headers.
The `real-ip` plugin reads client IP from trusted proxies' headers.
There is no plugin or built-in mechanism for the standardized `Forwarded` header per RFC 7239.

== Strengths

* High performance (Nginx + LuaJIT foundation, very low latency)
* Fully dynamic configuration (no restarts; millisecond propagation via etcd watch)
* Rich plugin ecosystem (78 built-in plugins)
* Multi-language plugin support (Lua, Go, Java, Python, Node.js, WASM)
* Cloud-native (stateless data plane, Kubernetes-native ingress controller)
* Active ASF project (16.3k stars, 491 contributors)
* AI Gateway capabilities (LLM proxying, token-based rate limiting since 2025)
* Protocol breadth (HTTP/1.1, HTTP/2, gRPC, WebSocket, MQTT, TCP/UDP, Kafka)
* All 78 plugins included in open-source edition

== Weaknesses

* etcd dependency in traditional mode adds operational complexity
* Lua as primary language has a smaller developer pool than Go/Java
* HTTP/3 still experimental (client-side only, known CVEs)
* No native RFC 7239 support
* Dashboard project has had slower development than the core gateway
* Documentation gaps for some advanced topics
* Inherits some Nginx limitations

== Open-Source vs. Enterprise

The gateway itself is *genuinely open source* under Apache 2.0 with all 78 plugins included.

API7 Enterprise (commercial offering by API7.ai) adds:

[cols="1"]
|===
| Enterprise-Only Features

| RBAC and workspace management
| Gateway groups (multi-tenancy)
| API portal with monetization
| GraphQL rate limiting and proxy cache
| SOAP protocol conversion
| Secrets management (Vault integration)
| OAS validator
| GDPR/FIPS/SOC2 compliance
| 99.9% SLA and dedicated support
|===

No core proxy or routing features are withheld from the open-source edition.
