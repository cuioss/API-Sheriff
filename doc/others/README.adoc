== Top Three Contenders

Based on the evaluation, three gateways stand out as the closest matches:

. *Apache APISIX* -- link:apache-apisix.adoc[Fact Sheet]
. *Traefik Proxy* -- link:traefik.adoc[Fact Sheet]
. *KrakenD* -- link:krakend.adoc[Fact Sheet]

The remaining three (link:kong.adoc[Kong], link:tyk.adoc[Tyk], link:gravitee.adoc[Gravitee]) were excluded from the top tier due to OSS edition limitations (Kong frozen at 3.9.1), mandatory external dependencies (Tyk requires Redis), or heavy infrastructure requirements (Gravitee requires MongoDB + Elasticsearch).

== Key Gaps (no gateway covers)

[cols="1,2"]
|===
| Requirement | Status

| RFC 7239 Forwarded header
| Not supported by any evaluated gateway. All use legacy `X-Forwarded-*` headers only.

| Step-up authentication (RFC 9470)
| Not supported by any evaluated gateway. No gateway parses upstream `WWW-Authenticate` challenges to trigger re-authorization.

| Distributed configuration (dynamic discovery via `/api/gateway-config.json`)
| Not natively supported by any evaluated gateway. Custom implementation required.
|===

== Recommendation

*Apache APISIX* covers the most manifest requirements in its open-source edition, particularly OIDC Relying Party, token mediation, and automatic token refresh -- features that are enterprise-only in Kong and Traefik.

However, *no existing gateway* satisfies all mandatory requirements.
The strongest gaps are RFC 7239, RFC 9470 step-up authentication, and the distributed configuration discovery pattern -- all of which require custom implementation regardless of the chosen base.
This confirms the rationale for building API Sheriff as a purpose-built gateway.
