= Variant 2 -- BFF, Session-based
:toc:
:toclevels: 3
:sectnums:

== Summary

In the session-based Backend-For-Frontend variant, API Sheriff is a full *OIDC confidential
client (Relying Party)*. It runs the authorization-code + PKCE login flow on behalf of a
browser-based SPA, holds the resulting tokens *server-side*, and gives the browser only an
opaque *session-id cookie*. On each API call the gateway looks up the stored token by session
id and injects it as `Authorization: Bearer` into the upstream request.

This is manifest *Variant B*. It is *stateful* -- it requires a server-side session/token
store -- but provides strong isolation: the token never leaves the server, not even in
encrypted form. This is the variant to start with; the link:03-bff-cookie.adoc[cookie-based
variant] is the stateless evolution.

NOTE: This variant is the *stateful BFF* shape of `CLIENT-19` (opaque session-id cookie +
server-side token store, backend proxies resource calls). The authoritative specification,
its precedence over the archived manifest, and the `token-sheriff-client` engine dependency
(the engine performs the OAuth legs; API Sheriff owns the browser-facing BFF side) are stated
canonically in
link:../architecture.adoc#_authentication_layer[Architecture -- Authentication Layer].

== Login Flow

The login handshake is shared with the link:03-bff-cookie.adoc[cookie-based variant]; only
the final binding step differs. The gateway redirects the unauthenticated browser to the IdP
with PKCE, `state`, and `nonce`; after the user authenticates, the IdP redirects the browser
back to the gateway callback with the authorization code as a query parameter (a `302`, not a
POST); the gateway -- driving the `token-sheriff-client` engine -- verifies `state` and issuer, exchanges the
code server-to-server, validates the tokens offline, and -- in this variant -- *stores them
server-side* and sets an opaque session-id cookie.

image::../resources/diagrams/bff-login-sequence.svg[OIDC confidential-client login: redirect to IdP, code exchange, token validation, session binding, align=center]

== Steady-State API Call

Once the session exists, an API call carries only the session-id cookie. The gateway:

. Validates the session cookie and resolves the session id.
. Looks up the stored access token in the server-side store.
. If the token is near expiry, transparently refreshes it (see below).
. Injects `Authorization: Bearer <token>` and forwards, applying the same
  link:01-base-gateway.adoc[base-gateway] filtering and zero-trust forward policy.

image::../resources/diagrams/bff-token-mediation.svg[Session-based versus cookie-based token mediation compared side by side, align=center]

The left-hand column above is this variant; the token lives in a server-side store keyed by
the session id.

== Token Storage

[cols="1,3"]
|===
| Aspect | Design

| Store
| A server-side store keyed by session id, holding the access token, refresh token, the raw
  ID token (needed as `id_token_hint` at logout), and token metadata (expiry, `acr`,
  `auth_time`, `sid`). The store maintains a secondary index by `sid`/`sub` so a back-channel
  logout token destroys the matching session in O(1) -- never a full scan. Configured as
  `oidc.session.store`; *`memory` is the only initial option*, sufficient for single-node or
  sticky-session deployments. A multi-node non-sticky deployment would need a shared store,
  which is deliberately unsupported for now (below).

| Trade-off
| A shared store (e.g. Redis) reintroduces an external dependency -- exactly what the
  link:../features-analysis.adoc#_features_to_avoid[feature analysis] warns against. The
  link:03-bff-cookie.adoc[cookie-based variant] exists to avoid this: it is stateless and needs
  no store. Choose session-based only where server-side-only token custody is a hard requirement.

| Session cookie
| `HttpOnly`, `Secure`, `SameSite=Lax`, `__Host-` prefix; carries only the opaque id, never
  token material. CSRF defence on state-changing calls is the BFF's responsibility (below).
|===

[[_csrf_defence]]
== CSRF Defence

Per `CLIENT-19`, backend-API CSRF is the BFF's concern (the callback endpoint itself is already
CSRF-protected by the session-bound `state`, `CLIENT-5`). Because the browser authenticates with
a cookie, every `auth.require: session` route applies *origin verification* on unsafe methods
(everything except GET/HEAD/OPTIONS):

* The request must carry an `Origin` header whose value is in
  `oidc.session.csrf.trusted_origins` -- the list of origins the application is served from,
  defaulting to the origin of `oidc.redirect_uri`. The `Origin` header names the *calling
  page's* origin, so an SPA served from `https://app.example.com` calling a gateway at
  `https://gw.example.com` must have `https://app.example.com` listed (see
  link:../configuration.adoc#_oidc[`session.csrf.trusted_origins`]). Comparison is exact,
  full-origin (scheme + host + port).
* If `Origin` is absent, `Sec-Fetch-Site: same-origin` is accepted (that header is an enum --
  `same-origin` / `same-site` / `cross-site` / `none` -- not an origin, so no origin comparison
  applies). Anything else -- a non-trusted `Origin`, `Sec-Fetch-Site: cross-site`, or neither
  header present -- is rejected with `403` and a `CSRF_REJECTED` event.
* The session cookie is set `SameSite=Lax` as defence-in-depth -- it blocks cross-site POSTs in
  modern browsers on its own, but origin verification remains the enforced check (older
  browsers, subdomain edge cases).

This is fixed behaviour, not configuration: a cookie-authenticated gateway without CSRF defence
is simply broken, so there is no knob to disable it.

NOTE: CSRF and CORS are the two halves of the cross-origin story. `trusted_origins` governs
what the *gateway accepts*; for an SPA served from a different origin than the gateway, the
browser additionally requires `security_headers.cors` enabled with `allow_credentials: true`.
This holds only for *same-site* origins (same registrable domain, e.g. `app.example.com` ->
`gw.example.com`): the cookie is fixed at `SameSite=Lax`, so a truly cross-*site* SPA never
gets it attached at all -- host the app under the gateway's site. Same-origin deployments need
neither entry.

== Logout

Logout follows `CLIENT-13` (RP-initiated logout) with the browser-facing parts owned by the
gateway as BFF:

. The browser calls `oidc.logout.path` (default `/logout`). The gateway *revokes* the stored
  access and refresh tokens at the IdP (RFC 7009, via the engine's revocation transport),
  *destroys* the server-side session, expires
  the session cookie, and redirects the browser to the IdP `end_session_endpoint` with
  `id_token_hint`, the pre-registered `post_logout_redirect_uri` (exact string match --
  open-redirect defence), and a *session-bound `state`* (`CLIENT-13`).
. The IdP returns the browser to `post_logout_redirect_uri` -- a *gateway-owned* path, so the
  gateway can *verify the returned `state`*. Because the session is already destroyed by
  step 1, the expected value lives in a short-lived (60 s), single-use
  `+__Host-sheriff-logout+` cookie set alongside the end-session redirect; the return leg
  compares against it, clears it, and redirects the browser on to
  `oidc.logout.final_redirect`.
. *Back-channel logout* (IdP-initiated, e.g. admin session revocation): the IdP posts a signed
  logout token to `oidc.logout.backchannel_path`. The gateway verifies the signature against
  the issuer's JWKS (reusing `token-sheriff-validation`'s issuer/JWKS infrastructure) and then
  applies the logout-token checks of OIDC Back-Channel Logout 1.0 itself -- `events` must
  contain the back-channel-logout event, `nonce` must be *absent*, `sub` and/or `sid` present --
  since the validation library has no logout-token pipeline; these checks are gateway code.
  On success it destroys the matching session. Because this variant holds sessions server-side,
  revocation is *immediately effective* -- an advantage over the
  link:03-bff-cookie.adoc[cookie-based variant].

== Transparent Token Refresh

When the stored access token is within the configured leeway of expiry, the gateway refreshes
it through the engine using the stored refresh token (with rotation), transparently during a
normal API request.
On refresh-token reuse detection the token family is revoked; on refresh failure the session
is destroyed and the request is handled as unauthenticated -- a *navigation* request is
re-driven through the login flow, while an API/XHR request receives `401`
`application/problem+json` (the same content negotiation as `require: session`). See
link:../configuration.adoc[`oidc.session.refresh`].

== Step-Up Authentication (RFC 9470)

Because the gateway holds the session, it can honour an upstream `401` carrying
`insufficient_user_authentication`: it parses the required `acr_values` / `max_age` / `scope`,
attempts a silent refresh, and -- if that cannot satisfy the challenge -- re-drives the
authorization-code flow with the elevated parameters, then replays the request. This is a
link:../features-analysis.adoc#_differentiators[differentiator] no competitor covers.

== Configuration

[source,yaml]
----
# gateway.yaml
oidc:
  issuer: https://idp.example.com/realms/main
  client_id: api-sheriff
  client_secret: ${SHERIFF_CLIENT_SECRET}
  redirect_uri: https://gw.example.com/auth/callback
  scopes: [openid, profile, orders.read]   # must include every scope a route requires
  session:
    mode: server                 # server-side token store (this variant)
    store: memory                # the only initial option (single-node / sticky sessions)
    cookie_name: __Host-sheriff
    ttl_seconds: 3600
    refresh: { enabled: true, leeway_seconds: 30, on_failure: reauthenticate }
----

[source,yaml]
----
# endpoints/app1.yaml           (base_url APP1 lives in topology.properties)
endpoint:
  id: app1
  base_url: APP1
  auth: { require: session, required_scopes: [orders.read] }  # default for all routes here
  routes:
    - id: orders-api
      match: { path_prefix: /api/orders }
      forward:
        headers_allow: [accept, content-type]
        # the mediated "Authorization: Bearer" is injected automatically on session routes
      upstream: { path: /orders }
----

== Properties

* *Stateful* -- requires a server-side token store.
* *Strong isolation* -- the token never leaves the server in any form.
* *Full OIDC RP* -- confidential-client login, transparent refresh, RFC 9470 step-up.
